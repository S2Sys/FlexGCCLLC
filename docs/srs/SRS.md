3. Two-Hour Assignment
Assignment
Build or describe a simple Work Request Tracker module for an existing platform built with ASP.NET Core Web APIs and Next.js. You do not need to build a fully production-ready app. Focus on clean structure, practical decisions, and the core backend/frontend flow.
Functional Requirements
A work request has:
•	Id
•	Title
•	Client name
•	Description
•	Priority: Low, Medium, High
•	Status: New, InProgress, Blocked, Completed
•	Due date
•	Created date
•	Updated date
•	Notes, optional
The user should be able to:
•	View a list of work requests.
•	Filter by status.
•	Search by title or client name.
•	Create a new work request.
•	Update the status of a work request.
•	Add a note to a work request.
Backend Requirements
Use ASP.NET Core Web API. Provide code or clear code snippets for the following:
•	Data model/entities.
•	Request/response DTOs where appropriate.
•	API endpoints.
•	Validation and error handling.
•	Persistence using an in-memory list, SQLite, SQL Server, or clear pseudo-code with a SQL schema.
GET    /api/work-requests?status=&search=&page=&pageSize=
GET    /api/work-requests/{id}
POST   /api/work-requests
PATCH  /api/work-requests/{id}/status
POST   /api/work-requests/{id}/notes
Show how you would handle missing required fields, invalid status, invalid priority, request not found, and a basic API error response format.
SQL Requirement
•	Provide a simple SQL schema for the module.
•	Include tables, primary keys, foreign keys if any, at least two useful indexes, and a short explanation of why those indexes help.
Frontend Requirements
Use Next.js / React. Provide code or clear code snippets for:
•	A page/component showing the work request list.
•	Status filter.
•	Search input.
•	Create request form or modal.
•	Status update action.
•	Loading and error states.
Architecture Note
Write a short note of 300-500 words covering your backend structure, frontend structure, trade-offs made because of the 2-hour limit, production improvements, what you would demo first, and remaining risks or assumptions.
Acceptance Criteria
Include 5-8 acceptance criteria. Example:
1. User can see a paginated list of work requests.
2. User can filter work requests by status.
3. User cannot create a request without title, client name, priority, status, and due date.
Known Issues / Not Done
Include a short list of anything you did not complete or would improve later. Example:
Known issues:
- Authentication is not implemented.
- Status transition rules are not enforced yet.
- Styling is minimal.
